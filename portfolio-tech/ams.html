Asset Management System

ASP .NET MVC / C# / SQL Server / Bootstrap / JQuery

Overview

The Asset Management System (AMS) is a platform developed to support asset management needs for companies, hospitals, and their branches across various cities in Indonesia. The system covers key processes such as asset registration, transfer, decommissioning, maintenance, and other related operations.

Role

Fullstack Developer

Tech Stack

ASP.NET MVC (C#)
Microsoft SQL Server
Bootstrap
JQuery

Background

The system was initially developed in early 2021 and was officially handed over to me in mid-2023. At that time, users were experiencing several issues with the existing AMS, including critical bugs, decreased performance, and a confusing user interface. In addition, users required new features to better support their operational needs.

With approval from upper management, I was granted the authority to implement major changes in order to adapt the system to meet the expectations of new users, while preserving the existing workflow. By implementing the six steps of Software Development Life Cycle (SDLC), the development was completed in early 2024, and I continue to maintain the system by fixing bugs and adding minor features to enhance user efficiency.

Planning

The further development of The Asset Management System (AMS) was started after hospital management, in coordination with the board of directors, agreed to enhance AMS with new features to support hospital operations in asset management.

The system scope includes asset registration, asset usage tracking, asset transfers, asset write-offs, asset loans, maintenance management, various asset-related reports, and user account management. However, the AMS currently has limitations—it's not yet integrated with external systems such as purchasing or accounting and is only available in a web version.

The development team consisted of 2 full-stack developers, 1 system analyst, and 1 supervisor. I served as one of the full-stack developers.

Analysis

The previous version of AMS had access control limited to branch-level and lacked unit/room-level access, which made it insufficient for the hospital's operational needs at the unit level. Role-based access was also limited to menu-level access and did not extend to action-level permissions. The AMS also lacked essential input fields required by the ASPAK (Ministry of Health Asset Reporting System), and asset transfers could only be conducted by asset managers, not by asset owners.

The users involved in the system include:

  • Admin: Asset managers with the highest level of authority.
  • Accounting Operator: Users with authority at the branch/company level.
  • Head of Unit: Users with authority at the room/unit level.
  • User: Individuals responsible for managing assets assigned to them.
  • Technician: Users responsible for addressing asset maintenance requests.
  • Superadmin: IT team members who manage all user accounts.

To improve asset management at the unit level, the hospital leadership requested the following features:

  • Room-level access rights
  • Role-based access for unit heads
  • Additional input fields required for ASPAK submissions
  • Asset transfer requests, both inter-room and inter-hospital
  • Asset write-off requests
  • Approval workflows for asset transfer and write-off requests

Beyond these requested features, my teammate and I identified several process flaws in the existing system and outdated technology, which required updates. We also initiated the integration of the authentication system with the company's Single Sign-On, so users no longer need separate credentials for AMS.

Design

The system redesign began with discussions involving the team and the IT supervisor to review the requested features and identify existing issues. We then created new database tables to support the required functionality and analyzed existing code to understand system workflows—this was crucial due to the lack of documentation. For complex workflows, we developed flowcharts for better comprehension.

The well-designed interfaces from the old system were retained in terms of layout but updated with a modern UI template to align with other internal systems, ensuring a unified branding. Less user-friendly interfaces were redesigned for improved usability. Our design principles emphasized consistency, ease of use, and strong branding for AMS.

Implementation

The system was built using the ASP .NET MVC framework with C# and integrated with Microsoft SQL Server. The front end used a premium Bootstrap 4 template with jQuery 3. Development tools included Visual Studio 2022, Visual Studio Code, Table Plus, and Figma.

Development was shared between two full-stack developers who divided tasks across modules and collaborated on code refactoring. My key contributions included:

  • Developing system branding (e.g., logo, layout structure, icons, illustrations)
  • Designing and creating database tables
  • Writing stored procedures for complex queries
  • Translating user requirements into CRUD functions, mostly accessible via REST APIs
  • Refactoring code and queries for efficiency and preventing timeouts
  • Adding detailed comments to complex code for easier maintenance
  • Coordinating with the IT supervisor and system analyst to ensure alignment with user requirements

Testing

Testing was conducted by the system analyst through periodic manual testing after each module was completed. Any reported bugs were immediately addressed by the developers. After internal testing was completed, a meeting was held with hospital leadership and asset managers to present the system. Feedback was mostly minor updates or additional features.

Deployment / Release

After completing all requested features and passing all tests, the updated AMS was officially launched in early 2024. Users were informed ahead of time to prepare for the system migration. The launch and migration proceeded smoothly. The system analyst provided training—both in person and virtually—to users across hospital branches and also prepared user documentation.

Maintenance

Ongoing maintenance continues, mainly addressing bug fixes, minor updates, and ensuring smooth user experience. In the future, AMS is planned to be rebuilt using a microservices architecture in Golang, integrated with the company's inventory / supply chain system to facilitate seamless conversion from procurement to asset registration.
