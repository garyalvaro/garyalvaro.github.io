SuperInventory (Supply Chain Management)

Go / PostgreSQL / MongoDB / RESTful API / Google Cloud Platform (GCP)

Overview

SuperInventory is a web-based internal system designed to streamline the company's supply chain and warehouse operations. Built as part of the company's long-term digital transformation roadmap, it consolidates multiple legacy systems into a single, scalable platform covering inventory tracking, purchasing, sales, distribution, and light manufacturing. The project focuses on improving data integrity, simplifying workflows, and enhancing system performance by adopting a microservices architecture. My role as a backend engineer involved developing robust APIs, optimizing data flow between services, and ensuring the platform could support diverse business operations across multiple subsidiaries.

Planning

The main objective of the SuperInventory system is to integrate multiple internal applications related to supply chain processes—including purchasing, stock logging, item issuing, sales, cashiering, shipping, and basic manufacturing. This centralized platform aims to improve data accuracy, streamline workflows, and support decision-making across business units.

System scope is limited to internal company use, covering the full distribution flow—from suppliers to warehouses, and from product usage to sales—while also supporting basic manufacturing operations. The system provides various stock-related reports and includes integration with the company's accounting system. However, the system does not support marketplace integration and is only available as a web-based application.

The project involves collaboration across multiple roles and departments, including:

  • Assistant of Manager (formerly Supervisor)
  • Assistant of Supervisor
  • Backend developers
  • Frontend developers
  • UI/UX designer
  • System analyst
  • End users from multiple subsidiaries, each with distinct operational workflows

Analysis

The previous system relied on outdated technologies and was deployed on a monolithic architecture hosted on on-premise servers. It was fragmented into multiple standalone applications, which led to operational inefficiencies, including data duplication, especially in product registrations, and inconsistent user experiences due to an interface that was not fully user-friendly.

The core requirements for the new backend system included:

  • Migrating from on-premise to cloud infrastructure to ensure scalability, reliability, and easier deployment.
  • Rebuilding the system using a microservice architecture, allowing for modular development, better fault isolation, and improved maintainability.
  • Designing a more unified and flexible database schema capable of supporting the diverse needs of supply chain operations, including purchasing, warehouse management, production, and distribution.

The system requirements were gathered through an in-depth review of legacy features and functionalities, supplemented by multiple meetings with end users across departments to understand practical challenges and gather improvement suggestions.

All APIs developed in this project were designed to be consumed by a web-based frontend application, making backend stability, clear data contracts, and performance optimization critical elements of the development process.

Design

The backend system was architected using a microservices approach, with each service exposed via RESTful APIs. The internal code structure followed a clear separation of concerns, organized into Controller–Service–Repository layers to promote maintainability and scalability.

The database schema was designed entirely from scratch, collaboratively developed and reviewed in regular meetings with the backend team. While the feature set referenced the legacy system, the new design was aimed at delivering more consistency, scalability, and adaptability across multiple supply chain domains.

API documentation was generated and maintained using Swagger, allowing frontend developers and stakeholders to easily reference endpoint specifications and payload structures.

In terms of security and authorization, several key mechanisms were implemented:

  • JWT (Bearer Tokens) for authenticating and identifying users accessing the APIs
  • API Keys for secure communication between internal microservices (service invocation)
  • Hashing and digital signature mechanisms for integrating with third-party/external APIs
  • SSH Keys for securely accessing cloud-hosted databases
  • Role-based access control (RBAC) to define and enforce user permissions at the application level

The design emphasized both flexibility and security to ensure smooth system operation across departments and business units with varying operational needs.

Implementation

The backend development was carried out using the Go programming language, leveraging PostgreSQL and MongoDB as the primary databases. The project was built on top of an in-house framework designed by the internal engineering team, utilizing a variety of Go libraries to support different functionalities.

Development tools included:

  • Visual Studio Code for code authoring
  • TablePlus for managing PostgreSQL databases
  • MongoDB Compass for visualizing and querying MongoDB collections
  • Swagger/Postman for APIs documentation

As a backend developer, I was involved in the implementation of a wide range of modules, including:

  • Product Master Data
  • Purchase Request
  • Purchase Order
  • Sales Order
  • Delivery Order
  • Assembly/Manufacturing
  • and many more

Beyond feature development, I was frequently assigned to resolve bugs across multiple modules, including:

  • Debugging and correcting business logic flaws from modules previously handled by team members who had left
  • Maintaining code consistency and stability across the platform

This rotating responsibility allowed me to gain broad exposure across nearly all modules within the SuperInventory system.

The project utilized built-in error-handling capabilities provided by the internal framework. System logging was occasionally reviewed via Google Cloud Platform (GCP) for diagnostics and monitoring. At this stage, the project did not implement automated unit or integration testing, and CI/CD pipelines were managed separately by the DevOps team.

My Role As Backend Developer

As a backend developer in the SuperInventory project, I was responsible for developing, maintaining, and optimizing critical API endpoints that power the company's internal supply chain management system. My contributions spanned across various modules, where I was actively involved not only in implementation but also in system design discussions, database modeling, and codebase restructuring.

One of the most technically fulfilling modules I handled was the Product Master, where I implemented a CQRS architecture that separated read operations (using MongoDB) and write operations (using PostgreSQL). This gave me hands-on experience with MongoDB queries and optimized read-heavy workloads. I also successfully integrated a Publisher-Subscriber (Pub/Sub) pattern in Golang, enabling real-time data synchronization between master data and transaction-level records across services.

Another significant responsibility was the integration of accounting transactions. Since nearly all operations in SuperInventory are linked to accounting (e.g., payables, receivables, journal entries, and cash flow), I was trusted to bridge our platform with the accounting system via secure API invocation and scheduled jobs. I worked with Google Cloud Platform logs to monitor and verify the successful execution of scheduled background processes.

I also developed the Assembly (Manufacturing) module from scratch — from process mapping to database design and API development. This feature allows users to combine multiple raw materials into finished goods, which required seamless handling of both stock deduction and product creation logic, including reporting integration.

In terms of code quality and collaboration, I played a key role in refactoring and globalizing functions that were previously duplicated across modules. By abstracting shared logic into core functions, the codebase became cleaner and less prone to bugs. This approach significantly improved maintainability and reduced debugging time for the team.

I was often assigned to maintain and extend legacy code left by previous developers, particularly the Purchase Order module. The biggest challenge was reverse-engineering undocumented logic and rewriting sections with clear inline comments to ensure readability and long-term maintainability.

Beyond coding, I took the initiative to mentor junior backend developers, introducing them to the company's tech stack, Git flow, deployment processes, and backend design principles like Controller–Service–Repository, Pub/Sub, and CQRS.

Cross-functional collaboration was also a core part of my role. I worked closely with system analysts to interpret user requirements, DevOps to assess load and performance, and frontend engineers to ensure API integration matched the expected UI behavior. I often helped debug and explain complex backend logic when misalignment occurred.

Throughout this project, I honed my ability to convert business requirements into scalable backend logic, improve system performance, and maintain clarity in code for both myself and my teammates. In the future, I hope to explore backend integrations with Google Services, WebSockets, Payment Gateways, and even AI-based tools to further enhance user productivity and experience.

Testing

Initial testing was performed directly by me as the backend developer to ensure that each API endpoint was free from syntax errors and followed the correct business logic flow. This step was essential before handing over features for broader testing.

Further testing was conducted by the frontend team and system analysts, who carried out:

  • Manual testing during the development phase to verify API integration with frontend components
  • User Acceptance Testing (UAT) just before the release to production, ensuring that the system met the business requirements and was usable by real users

Feedback was typically communicated in the form of written notes and annotated screenshots, which highlighted the status of each tested endpoint—whether it functioned correctly or still contained bugs. I responded promptly to these reports by debugging, making adjustments to the business logic, or improving the data structure if necessary.

Deployment / Release

The deployment process for the backend services was streamlined and efficient, thanks to the support of a dedicated DevOps team. My primary responsibility in this phase was to create and tag the final commit, signaling that the feature or fix was ready for deployment.

The backend was deployed to Google Cloud Platform (GCP), utilizing CI/CD pipelines integrated with Bitbucket for automated build and release processes. This workflow significantly reduced deployment friction and allowed the development team to focus on feature delivery and quality improvements.

  • Feature enhancements and minor upgrades requested by users and business units to improve usability or adapt to operational changes
